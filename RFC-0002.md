# Kotoba — Japanese Knowledge Catalog v1.0

Status: Accepted

Author: Guga + ChatGPT
Date: 2026-07-03

---

## 1. Purpose

This RFC defines the first version of the **Japanese Knowledge Catalog** for Kotoba.

The Knowledge Catalog is the canonical list of Japanese learning knowledge items used by:

* Knowledge Browser
* Lesson Browser
* Mastery Dashboard
* Wrong Question Analysis
* Review System
* Future AI Tutor

The Catalog answers one key question:

> What exactly can a learner know, partially know, misunderstand, forget, or master?

---

## 2. Key Decisions

### Decision 1: Knowledge Granularity

Accepted option: **A — fine-grained Knowledge items**

A Knowledge item should be small enough to be diagnosed independently.

Examples:

* `〜ながら`
* `〜ば`
* `〜たら`
* `〜なら`
* `〜ている（进行）`
* `〜ている（状态）`
* `小句 + 時`
* `动词た形 + 時`
* `ます形`
* `助词 は`
* `助词 が`

Do not use only broad concepts such as:

* 条件表达
* 连接表达
* 时态表达

Broad concepts should be represented as **Knowledge Groups**, not as individual Knowledge items.

Reason:

* AI error analysis needs precision.
* Dashboard needs actionable mastery tracking.
* A learner should know exactly which form is weak.

---

### Decision 2: Catalog Authority Source

Accepted option: **A — Standard Japanese Third Edition as the main path + JLPT requirements as supplement**

Primary source path:

```text
标准日本语第三版
→ 初级上
→ 初级下
→ 中级
```

Supplementary classification:

```text
JLPT N5
JLPT N4
JLPT N3
JLPT N2
JLPT N1
```

This means:

* The learner’s current study path follows 标准日本语第三版.
* Each Knowledge item can also be tagged with JLPT level.
* Future sources such as TRY, 新完全掌握, 真题, anime, news, podcasts can be mapped later.

## 3. Catalog Design Principles

### Principle 1: Knowledge First, Lesson Second

The Catalog should not be built as:

```text
Lesson 27
→ Grammar 1
→ Grammar 2
```

It should be built as:

```text
Knowledge Item
→ may appear in Lesson 27
→ may appear in JLPT N4
→ may appear in TRY N4
→ may appear in wrong questions
```

Lessons are learning routes.
Knowledge is the source of truth.

---

### Principle 2: One Knowledge, Multiple References

Example:

```text
〜ながら
→ 标准日本语第三版 初级下 第27课
→ JLPT N4
→ Connection Group
→ Similar: つつ
→ Confusing: 〜間に
```

---

### Principle 3: Group for Understanding, Item for Diagnosis

Use **Knowledge Groups** for learning structure.

Use **Knowledge Items** for mastery tracking.

Example:

```text
Knowledge Group: 条件表达
├── 〜と
├── 〜ば
├── 〜たら
└── 〜なら
```

A learner may understand the group concept but still be weak on `〜なら`.

---

### Principle 4: Lesson Mapping Must Be Versioned

Every lesson mapping must include:

```text
textbook_name
edition
book_level
lesson_number
source_status
```

Example:

```text
textbook_name: 标准日本语
edition: 第三版
book_level: 初级下
lesson_number: 27
source_status: verified_by_user
```

This prevents mixing knowledge points from different textbooks or editions.

## 6. Knowledge Item ID Rules

Each Knowledge item must have a stable ID.

Format:

```text
K-[DOMAIN]-[GROUP]-[ITEM]
```

Examples:

```text
K-GRA-CON-NAGARA
K-GRA-TIME-TOKI
K-GRA-MOD-DESHOU
K-GRA-CONDITION-BA
K-GRA-CONDITION-TARA
K-GRA-PARTICLE-WA
K-GRA-PARTICLE-GA
K-VOC-VERB-TABERU
K-KAN-CHAR-食
```

Recommended domain codes:

```text
GRA = Grammar
VOC = Vocabulary
KAN = Kanji
EXP = Expression
REA = Reading
LIS = Listening
PHO = Phonology
PRA = Pragmatics
```

## 7. Knowledge Item Schema

Each Knowledge item should follow this schema.

```json
{
  "id": "K-GRA-CON-NAGARA",
  "title": "〜ながら",
  "title_cn": "一边……一边……",
  "domain": "Grammar",
  "group": "Connection",
  "jlpt_level": "N4",
  "difficulty": 3,
  "status": "active",
  "description_cn": "表示同一主体同时进行两个动作，通常后项是主要动作。",
  "description_jp": "",
  "form": "动词ます形去ます + ながら",
  "concept": "同时进行",
  "usage_notes": [
    "前后动作通常为同一主体。",
    "后项动作通常是主要动作。",
    "前项动作常作为伴随动作。"
  ],
  "common_mistakes": [
    "前后主语不一致。",
    "把瞬间动作误用于ながら。",
    "与〜間に混淆。"
  ],
  "example_jp": "音楽を聞きながら、勉強します。",
  "example_cn": "一边听音乐，一边学习。",
  "source_status": "seed",
  "created_at": "2026-07-03",
  "updated_at": "2026-07-03"
}
```

## 8. Knowledge Relationship Schema

Knowledge relationships should be stored separately from Knowledge items.

```json
{
  "id": "REL-001",
  "from_knowledge_id": "K-GRA-CON-NAGARA",
  "to_knowledge_id": "K-GRA-VERB-MASU-STEM",
  "relationship_type": "prerequisite",
  "description": "〜ながら requires understanding of ます形 stem."
}
```

Supported relationship types:

```text
prerequisite
related
similar
contrast
confusing
next_learning
belongs_to_group
```

## 9. Lesson Mapping Schema

A lesson can contain multiple Knowledge items.
A Knowledge item can appear in multiple lessons.

```json
{
  "id": "MAP-SJ3-SHIMO-27-001",
  "knowledge_id": "K-GRA-TIME-TOKI",
  "textbook_name": "标准日本语",
  "edition": "第三版",
  "book_level": "初级下",
  "lesson_number": 27,
  "lesson_title": "子供の時、大きな地震がありました",
  "section": "grammar",
  "source_status": "verified_by_user"
}
```

Source status values:

```text
seed
verified_by_user
verified_by_textbook
needs_review
deprecated
```

## 10. Mastery Color System

Use this shared color logic across Dashboard, Knowledge Browser, and Lesson Browser.

```text
green       = 已稳定掌握
light_green = 较好
yellow      = 一般 / 不稳定
red         = 薄弱
dark_gray   = 未学习
blue        = 即将学习
```

Mastery score mapping:

```text
85–100 = 已稳定掌握
70–84  = 较好
50–69  = 一般
1–49   = 薄弱
0      = 未学习
future = 即将学习
```

## 14. Codex Implementation Notes

For SPRINT-001, Codex should not implement full database logic.

Use local seed JSON first:

```text
/data
├── knowledge.json
├── knowledge-groups.json
├── lessons.json
├── lesson-mappings.json
├── relationships.json
├── mastery.json
└── wrong-questions.json
```

The frontend should be built as if the data could later move to Supabase.

Do not hard-code UI values directly into React components.

---

## 15. Acceptance Criteria

SPRINT-001 can be considered successful if the Web MVP can:

1. Display a dashboard with mock mastery data.
2. Show Knowledge items grouped by domain and group.
3. Show Lesson 27 and its mapped Knowledge items.
4. Open a Knowledge Detail page for `〜ながら`.
5. Display relationships such as prerequisite, similar, confusing.
6. Use the mastery color system consistently.
7. Load all visible learning content from JSON seed data.
8. Keep all seed data editable without touching UI code.
